# 客服知识库问答系统最终提案

生成日期：2026-05-13

## 1. 结论

本系统应定位为“客服知识运营系统”，而不是简单 FAQ 或聊天机器人。它的核心目标是：把客服部门已有的文档、问答、人工经验沉淀为可检索、可审核、可追踪、可持续优化的知识资产，并通过 API 或 Skill 提供给客服人员使用。

最终推荐方案：

```text
客服工作台 / Skill / 第三方入口
        |
        | HTTPS API
        v
Node.js 后端服务（NestJS）
        |
        +-- 用户、权限、知识库、审核、日志、反馈
        +-- 检索编排：关键词检索 + 向量检索 + 重排 + 置信度策略
        +-- 文件解析、切片、向量化、索引任务
        |
        v
PostgreSQL + pgvector（MVP 主数据库）
        |
        +-- 业务数据强一致存储
        +-- 向量索引与全文检索
        +-- 后续可平滑拆出 Qdrant / Elasticsearch / Redis / MQ
```

数据库采用 **PostgreSQL + pgvector 单库起步**。理由是客服知识库早期最重要的是数据完整性、可管理性、事务一致性和低运维复杂度。向量数据本质上是知识文本的派生索引，MVP 阶段没有必要一开始引入独立向量数据库。系统设计上保留 `VectorStore` 抽象，未来当数据规模、并发或召回性能达到瓶颈时，再迁移到 Qdrant、Milvus 或 Elasticsearch 混合架构。

Skill 不能承载完整系统。Skill 只作为轻量接入层，负责收集问题、展示候选答案、提交客服选择或编辑结果；所有认证、权限、检索、知识沉淀、审核、日志和数据一致性都必须由后端服务处理。

## 2. 四份分析文档的共性

四个模型的分析文档在几个关键方向上高度一致：

1. 系统本质是 RAG + 知识库管理，而不是传统 FAQ。
2. 需要支持文档导入、解析、切片、向量化、检索、人工反馈闭环。
3. 结构化业务数据适合用关系型数据库，MongoDB 不是首选。
4. Skill 应该调用后端 API，不应该把完整后端写成 Skill 脚本。
5. 客服编辑后的答案不能直接污染正式知识库，应进入审核或版本流程。
6. Node.js 技术栈应优先考虑 NestJS / Express、Prisma、React 管理后台。
7. 开源项目可参考，但客服业务闭环通常需要定制。

这些共性是合理的，尤其是“后端独立服务 + Skill 调 API + 人工反馈审核入库”这条边界，必须作为架构原则固定下来。

## 3. 对分歧点的判断

### 3.1 PostgreSQL + pgvector 还是 PostgreSQL + Qdrant

两种方案都合理，但适用阶段不同。

MVP 和内部客服场景推荐 `PostgreSQL + pgvector`：

- 一个数据库管理事务、备份、迁移、权限和审计，运维简单。
- FAQ、文档、用户、审核、日志之间有关联关系，关系型数据库更自然。
- 可以在 SQL 中同时做状态过滤、租户过滤、权限过滤、全文检索和向量排序。
- 数据一致性更容易保证。

当出现以下情况，再拆出 Qdrant 或 Milvus：

- 知识 chunk 达到百万级以上。
- 查询并发明显增加，pgvector 成为瓶颈。
- 需要多向量召回、复杂 payload 过滤、分布式扩展。
- 需要独立扩缩容向量检索能力。

最终提案不是否定向量数据库，而是采用“单库起步、接口隔离、可替换演进”的路线。

### 3.2 是否二开开源项目

可以参考和验证，但不建议一开始深度二开大型通用产品。

当前可参考方向：

- FastGPT：TypeScript 技术栈，知识库、QA 拆分、混合检索、重排、OpenAPI 和工作流能力较贴近需求。适合做产品能力参考或快速 POC。需注意其开源协议对 SaaS 和商用展示的限制。
- AnythingLLM：Node/React/Express，MIT，支持多用户、文档管线、向量库和 Developer API。适合验证通用文档问答能力，但客服候选答案选择、审核、知识闭环仍需定制。
- Flowise：Node/TypeScript，适合可视化搭建 RAG 或 Agent 流程，适合原型和内部流程编排，不适合作为客服知识运营主系统。
- Dify：能力成熟，但后端主要是 Python，若坚持 Node.js 主栈，二开成本较高。

建议先用 FastGPT 或 AnythingLLM 做一周 POC，验证文件导入、召回质量、混合检索和 API 形态。正式系统仍建议自研业务层，避免被通用产品的数据模型和交互流程锁死。

## 4. 目标用户与核心场景

### 4.1 角色

- 管理员：维护客服账号、角色、知识库、审核流程、系统配置。
- 知识管理员：导入文档、维护标准问答、审核客服沉淀内容、关闭过期答案。
- 客服人员：输入问题、查看候选答案、选择答案、编辑答案、提交反馈。
- 系统调用方：Skill、内部客服系统、IM 机器人、浏览器插件等。

### 4.2 核心场景

1. 管理员导入历史知识文件：txt、md、docx、xlsx、csv，后续可扩展 pdf、网页、API。
2. 系统解析文件，识别 FAQ 或切成文档片段，生成 embedding，建立索引。
3. 客服输入问题，系统返回一个高置信答案或多个候选答案。
4. 客服选择已有答案，系统记录命中、使用、置信度和上下文。
5. 客服编辑或新增答案，系统保存为待审核知识。
6. 知识管理员审核后，正式发布为可检索知识。
7. 管理员可关闭、归档、替换、回滚知识版本。

## 5. 系统架构

### 5.1 推荐项目结构

```text
question-and-answer/
  apps/
    api/                  # NestJS 后端 API
    web/                  # React 管理后台与客服工作台
  packages/
    qna-core/             # 问答领域核心逻辑，不依赖 Web 框架
    shared/               # DTO、类型、错误码、权限常量
  prisma/
    schema.prisma
    migrations/
  docker/
    docker-compose.yml
```

`qna-core` 是业务核心包，不是外部 Skill。它封装检索、确认、反馈、审核、发布等领域逻辑。`apps/api` 负责 HTTP、认证、权限、请求校验、事务边界和调用核心服务。外部 Skill 通过 HTTPS API 调用 `apps/api`。

### 5.2 后端模块

- Auth 模块：登录、JWT、刷新令牌、密码策略、API Token。
- User 模块：用户、角色、部门、禁用状态。
- Knowledge 模块：知识条目、版本、标签、分类、启停、归档。
- Document 模块：文件上传、解析任务、来源追踪。
- Ingestion 模块：清洗、切片、FAQ 提取、embedding、索引构建。
- Search 模块：关键词检索、向量检索、融合排序、重排、置信度判断。
- Feedback 模块：客服选择、编辑、无效标记、未命中问题。
- Review 模块：待审核知识、发布、驳回、版本回滚。
- Audit 模块：操作日志、数据变更日志、调用日志。
- Integration 模块：Skill API、内部系统 API、Webhook。

## 6. 数据设计

### 6.1 核心原则

1. PostgreSQL 是业务事实源，向量索引是可重建的派生数据。
2. 正式知识必须有版本，不允许无痕覆盖。
3. 客服编辑内容默认进入待审核，不能直接污染正式知识库。
4. 删除优先使用软删除或归档，保留审计链路。
5. 外部模型调用和向量化任务使用异步任务状态，不放在数据库事务里长时间等待。

### 6.2 主要表

```text

  id
  filename
  file_type
  storage_key
  parse_status      # uploaded / parsing / parsed / failed
  checksum
  uploaded_by
  created_at

document_chunks
  id
  document_id
  knowledge_item_id
  chunk_index
  content
  token_count

  id
  knowledge_item_id

```

### 6.3 一致性策略

- 知识发布在数据库事务中完成：写入版本、更新 `knowledge_items.current_version_id`、更新状态。
- embedding 生成采用异步任务：事务提交后写入待处理任务，任务完成后更新 `embeddings.status=ready`。
- 使用 outbox pattern 记录“需要生成/刷新向量”的事件，避免业务数据已提交但索引任务丢失。
- 查询只使用 `active` 且 `embedding.ready` 的知识。若 embedding 失败，仍可通过关键词检索兜底。
- 关闭或归档知识时，将相关 embedding 标为 `stale` 或通过状态过滤排除，不物理删除。
- 所有关键操作写 `audit_logs`，支持追责、回滚和数据修复。

## 7. 检索与问答流程

### 7.1 知识导入流程

```text
上传文件
  -> 计算 checksum，避免重复导入
  -> 保存 documents 记录
  -> 异步解析文件
  -> Excel/CSV 优先识别为 FAQ
  -> txt/md/docx 按标题、段落、长度切片
  -> 可选：LLM 辅助抽取 FAQ，但结果必须待审核
  -> 写入 knowledge_items / knowledge_versions / document_chunks
  -> 创建 embedding 任务
  -> embedding ready 后进入可检索状态
```

切片建议：

- FAQ：问题和答案作为一个整体生成问题向量，可额外生成答案向量。
- 文档片段：按 300 到 800 中文字切片，保留 50 到 100 字 overlap。
- Excel：如果存在“问题/答案”列，直接导入 FAQ；无法识别列时进入人工映射步骤。

### 7.2 客服查询流程

```text
客服提问
  -> 请求鉴权和权限过滤
  -> 文本标准化
  -> 关键词/全文检索召回 TopN
  -> pgvector 语义检索召回 TopN
  -> 融合排序
  -> 可选 rerank
  -> 按置信度返回：
       高置信：返回首选答案 + 候选
       中置信：返回多个候选让客服选择
       低置信：返回未命中，允许客服编辑新答案
  -> 记录 search_session
```

### 7.3 反馈闭环

```text
客服选用已有答案
  -> 记录 selected
  -> 增加使用统计
  -> 不改变正式知识

客服编辑答案
  -> 记录 edited
  -> 生成 pending_review 知识版本
  -> 管理员审核
  -> 审核通过后发布并生成 embedding

客服标记无效或未命中
  -> 记录 rejected / no_answer
  -> 进入知识缺口列表
  -> 知识管理员补充
```

## 9. 高可用与可扩展设计

### 9.1 高可用

- API 服务无状态部署，支持多实例和负载均衡。
- PostgreSQL 使用主从或托管高可用实例，定期备份并演练恢复。
- 文件上传使用对象存储或兼容 S3 的存储，数据库只存 metadata 和 storage key。
- embedding、文件解析、索引任务通过队列异步处理，失败可重试。
- 所有外部模型调用设置超时、重试、熔断和降级。

### 9.2 可扩展

- `VectorStore` 抽象隔离 pgvector、Qdrant、Milvus。
- `EmbeddingProvider` 抽象隔离 OpenAI、Azure OpenAI、硅基流动、本地 bge 等模型。
- `DocumentParser` 插件化支持新增文件类型。
- 检索流程采用 pipeline 设计，便于插入 rerank、规则过滤、业务权重。
- 多租户可以通过 `tenant_id` 扩展到所有核心表和检索过滤条件。

### 9.3 数据完整性

- 所有外键关系由数据库约束保障。
- 关键状态使用枚举或 check 约束。
- 文件导入使用 checksum 防重复。
- 知识版本不可变，发布只是切换 current version。
- 审核、启停、归档必须写审计日志。

### 9.4 数据一致性

- 正式知识状态变化与版本发布在单个事务内完成。
- 向量索引允许最终一致，但必须可重建。
- 查询时通过状态过滤避免读到未审核、已关闭或索引过期的数据。
- 定期跑一致性巡检任务：检查 active 知识是否都有 ready embedding，检查 stale embedding 是否被排除。

## 10. 安全与权限

- 所有 API 默认鉴权，Skill 使用 API Token 或 OAuth/JWT。
- RBAC 权限：管理员、知识管理员、客服人员、只读审计员。
- 客服只能查询可见知识库，不能直接发布正式知识。
- 管理操作需要审计日志。
- 文件上传限制大小、类型、扩展名和 MIME，解析任务隔离执行。
- 敏感配置使用环境变量或密钥管理，不写入代码仓库。
- 对外 API 做限流，防止滥用 embedding 和 LLM 成本。

## 11. 技术选型

### 11.1 MVP 推荐栈

```text
后端：Node.js + TypeScript + NestJS
ORM：Prisma
数据库：PostgreSQL 16 + pgvector
队列：BullMQ + Redis（MVP 可先用数据库任务表，二期引入 Redis）
前端：React + Ant Design
文件解析：mammoth、xlsx、remark/markdown、纯文本解析
检索：PostgreSQL full-text / trigram + pgvector
模型：可配置 OpenAI / Azure OpenAI / 国内兼容 OpenAI API 的模型服务
部署：Docker Compose 起步，后续 Kubernetes
```

### 11.2 不推荐项

- 不推荐 MongoDB 作为主库。该系统有强关系、审核、版本、权限、日志、事务要求，MongoDB 优势不明显。
- 不推荐一开始拆多个数据库。除非已有明确规模压力，否则会增加一致性和运维成本。
- 不推荐把 Skill 写成完整后端。Skill 是入口，不是业务系统。
- 不推荐客服编辑结果直接发布。必须经过审核或至少进入待发布状态。

## 12. 分阶段实施路线

### Phase 0：开源项目验证，1 周

目标：验证现成 RAG 产品能力边界。

- 本地部署 FastGPT 或 AnythingLLM。
- 上传真实客服文档样本。
- 测试 FAQ、文档片段、多候选召回、API 调用。
- 记录召回质量、文件解析质量、二开难点。

产出：是否借鉴其数据处理策略、是否复用部分组件、正式系统的检索参数基线。

### Phase 1：MVP，2 到 4 周

目标：客服可以上传 Excel/Markdown/Docx，提问并获得候选答案，编辑后进入审核。

- 搭建 NestJS + Prisma + PostgreSQL + pgvector。
- 实现用户登录和基础 RBAC。
- 实现文件上传、解析、FAQ 导入、文档切片。
- 实现 embedding 任务和 pgvector 检索。
- 实现客服查询、候选展示、选择、编辑反馈。
- 实现知识列表、启停、审核发布。

### Phase 2：生产可用，4 到 8 周

目标：可稳定服务客服团队。

- 引入全文检索 + 向量检索融合排序。
- 引入操作审计、调用日志、数据质量报表。
- 引入队列、失败重试、任务监控。
- 增加知识版本回滚、批量导入、重复检测。
- 增加 API Token 和 Skill 接入。
- 完善备份恢复、监控告警、限流。

### Phase 3：智能增强，按效果推进

目标：提升准确率和运营效率。

- 引入 rerank 模型。
- 引入 LLM 自动抽取 FAQ，但必须人工审核。
- 引入知识缺口分析、低命中问题聚类。
- 引入多租户、多知识库权限。
- 当 pgvector 成为瓶颈时，迁移到 Qdrant 或 Milvus。

## 13. 验收标准

MVP 验收标准：

- 支持上传 txt、md、docx、xlsx 中至少三种格式。
- Excel FAQ 可以直接导入为标准问答。
- 客服提问后可返回 Top 5 候选答案。
- 客服可选择已有答案或编辑新答案。
- 编辑后的答案进入待审核，不直接发布。
- 管理员可审核、启用、关闭知识。
- 所有查询、选择、编辑、审核动作都有日志。
- 停用知识不会被检索返回。
- embedding 失败不影响业务数据完整性，可重试。

生产验收标准：

- API 服务可多实例部署。
- 数据库有备份和恢复方案。
- 文件解析和向量化任务失败可追踪、可重试。
- 查询接口有鉴权、限流、审计。
- 知识版本可回滚。
- 检索结果能展示来源，便于客服判断可信度。
- 有知识缺口报表和低命中问题列表。

## 14. 最终建议

最终路线应采用“自研业务系统 + 借鉴开源 RAG 产品 + 保留替换能力”的方案：

1. 不把系统做成单纯 Skill，Skill 只通过 API 调用后端。
2. 不一开始上复杂多数据库，MVP 使用 PostgreSQL + pgvector。
3. 不直接深度二开大型通用聊天产品，先用 FastGPT / AnythingLLM 做 POC，正式系统自研客服业务闭环。
4. 必须把审核、版本、审计、状态过滤、索引重建作为一等设计，不然后期知识会失控。
5. 检索不要只做向量相似度，应该做 FAQ 精准匹配、全文检索、向量检索、融合排序和置信度策略。

这套方案能在早期控制复杂度，同时保留面向企业级场景的高可用、可扩展、数据完整性和数据一致性演进空间。
