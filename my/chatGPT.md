# 二、你的核心架构（非常推荐）

我先给你一个我认为最合理的架构：

```text
                ┌────────────────┐
                │ Web 管理后台    │
                └──────┬─────────┘
                       │
                REST/GraphQL
                       │
        ┌──────────────┴──────────────┐
        │                             │
┌───────▼────────┐          ┌────────▼────────┐
│ 业务后台 API    │          │ AI/RAG 服务      │

└───────┬────────┘          └────────┬────────┘
        │                             │
        │                             │
        │                    Embedding/LLM
        │                             │
 ┌──────▼──────────┐         ┌───────▼─────────┐
 │ PostgreSQL      │         │ Vector DB       │
 │ 用户/FAQ/权限等 │         │ pgvector/Qdrant │
 └─────────────────┘         └─────────────────┘
```

重点：

# 不要把 Skill 当项目本体

# 三、关于数据库：到底怎么选？

这是你问得最关键的地方。

---

# 方案一（我最推荐）

# PostgreSQL + pgvector

也就是：

* 一个数据库解决：

    * 关系型数据
    * 向量搜索

这是目前最适合中小型 AI 项目的方案。

---

# 为什么推荐？

因为你需要：

## 1. 关系数据

例如：

* 用户
* 客服
* 权限
* FAQ
* 标签
* 审核状态
* 日志
* 命中统计

这些都属于强关系数据。

Mongo 并不擅长。

---

## 2. 语义搜索

你还需要：

```text
“退款多久到账”
≈
“退款什么时候到”
```

这需要：

* embedding
* 向量搜索

pgvector 就能做。

---

# 所以：

# PostgreSQL + pgvector

已经能覆盖 80% 需求。

而且：

* 运维简单
* 数据一致性强
* 不用维护多个数据库
* 开发效率高

---

# 什么时候再引入 ES / Qdrant？

当你：

* 百万级知识
* 高并发
* 海量检索
* 多租户 SaaS

再考虑：

| 组件            | 用途    |
| ------------- | ----- |
| Elasticsearch | 关键词搜索 |
| Qdrant        | 向量搜索  |
| Redis         | 缓存    |
| Kafka         | 异步任务  |

但你现在没必要。

---

# 五、真正的核心：不要只做 FAQ

你真正应该做：

# FAQ + 文档 RAG 混合

什么意思？

---

# FAQ 命中（高精度）

适合：

```text
退款多久到账？
```

直接精准命中。

---

# 文档检索（RAG）

适合：

```text
欧洲仓订单超过30天怎么处理？
```

这种问题：

* FAQ 未必有
* 但文档里可能有

所以：

# 你需要双路检索

---

# 最佳方案：

```text
用户问题
   ↓

① FAQ 精准匹配
   ↓
命中？
   ↓ yes -> 返回

no
   ↓

② 向量语义检索
   ↓

③ 返回 TopK 文档片段

④ LLM 总结答案

⑤ 客服确认/编辑

⑥ 保存为新 FAQ
```

这个设计非常关键。

因为：

# 人工修正 = 知识沉淀

你的系统会越来越聪明。

# 六、知识如何导入（这里很多人做错）

你说：

* txt
* md
* word
* excel

这是对的。

但重点不在“上传”。

重点在：

# 如何结构化

---

# 正确流程：

```text
上传文件
   ↓

文档解析
(docx/pdf/excel)

   ↓

文本切块
(chunking)

   ↓

AI 自动提取 FAQ
(可选)

   ↓

生成 embedding

   ↓

存储
```

---

# Chunk 很关键

不要整个文档 embedding。

应该：

```text
每 300~500 字
一个 chunk
```

否则召回会很差。

---

# Excel 特殊处理

Excel 常见是：

| 问题 | 答案 |
| -- | -- |

这时：

直接转 FAQ。

# 七、客服修正机制（这是产品灵魂）

你已经想到：

> 如果多个答案，客服重新编辑一个更合适的

这个非常对。

因为：

# AI 最怕脏知识

所以：

客服修正后的答案：

不能直接覆盖。

应该：

```text
客服编辑
   ↓
待审核
   ↓
管理员通过
   ↓
正式知识库
```

否则知识会越来越乱。

* 很短

* 数据库
* AI
* 审核

都写进 skill。
